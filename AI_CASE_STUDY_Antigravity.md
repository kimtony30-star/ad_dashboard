# 🛠️ Manus로 틀 잡고 Antigravity로 완성! 토큰은 아끼고 효율은 높이는 AI 코딩 도구 분산 활용법

## 📝 한줄 요약
단일 AI 도구(Manus)에만 의존하던 대시보드 프로젝트를 로컬 최적화 AI 도구인 Antigravity로 마이그레이션하면서, 토큰 소모량은 획기적으로 낮추고 빠르고 안전한 로컬 기반 업무 대시보드를 완성했습니다.

**바쁘시면 이것만 읽어도 돼요:**
- **의도된 가설 검증**: Manus는 넓고 방대한 초기 틀을 잡는 데 쓰고, 빠르고 토큰 가성비가 필요한 로컬 작업은 Antigravity로 분산 처리하는 하이브리드 전략 실험
- **스스로 디버깅하는 AI**: SQLite 데이터 적재 중 발생한 에러(NOT NULL constraint)를 프런트엔드-백엔드 구조를 분석해 알아서 원인을 찾고 치유함
- **뛰어난 아키텍처 이해력**: 기존 Manus 코드베이스를 넘겨주었음에도 Drizzle ORM 변환과 다크 테마 UI 구조를 찰떡같이 이해하고 적용함
- **결과**: 데이터 유출 걱정 없는 나만의 독립된 로컬 대시보드 환경 확보, "AI 도구는 환경에 맞게 쪼개 쓰는 것이 정답!"

## 🎯 이런 분들께 도움돼요
- Cursor, Manus 등 여러 AI 코딩 도구를 써보며 "생각보다 요금이 너무 많이 나오는데?" 고민하셨던 분들
- 사내의 민감한 데이터(대용량 CSV 등)를 클라우드 플랫폼에 올리기 껄끄러워 완전히 내 컴퓨터에서 구동되는 로컬 툴이 필요하신 분들
- 하나의 완벽한 AI 도구를 찾기보다 목적에 맞게 여러 AI 도구를 똑똑하게 조합하여 효율을 극대화하고 싶은 분들

## 😫 문제 상황 (Before)

최근 Manus를 이용해 '광고통계 대시보드'를 성공적으로 구축했지만, 제 마음속에는 개발자적인 호기심이 계속 존재했습니다. 
"수만 건의 대용량 데이터를 처리하고 유지보수할 때 플랫폼 종속적인 에디터를 계속 쓰는 게 맞을까?"
"Manus는 편리하지만 그만큼 토큰(Token) 사용량이 높은데, 로컬 환경에서 더 비용 효율적으로 구동되는 다른 AI는 어떨까?"

특히, 회사 내부망이나 내 컴퓨터 안(로컬 환경)에서 마음 편하게 데이터를 다룰 수 있는 **'가볍고 토큰 가성비 좋은 AI 환경'**에 대한 갈증이 컸습니다. 
그래서 기존에 완성된 Manus 코드베이스를 뚝 떼어다가, 최근 관심 있게 지켜보던 **'Antigravity(안티그래비티)'**에 던져주고 완전히 로컬 기반의 SQLite 환경으로 이관해 보는 테스트(가설 검증)를 시작하게 되었습니다.

## 🛠️ 사용한 도구
- **마이그레이션 및 고도화 도구**: Antigravity
- **특이사항**: 기존 Manus 환경에서 파생된 코드를 분석한 후, 로컬 기반(SQLite + Drizzle ORM)의 아키텍처로 변환하고 다크 테마 UI를 입히는 고도화 작업 수행

---

## 🔧 작업 과정

### 1단계: 낯선 코드베이스도 찰떡같이 이해하는 아키텍처 흡수력

먼저 기존 Manus에서 돌아가던 대시보드 코드들을 Antigravity 환경에 세팅하고 마이그레이션 방향을 지시했습니다.

```text
기존 Manus 환경에서 개발하던 광고통계 대시보드를 Antigravity 기반 플랫폼으로 마이그레이션해줘. 
백엔드 데이터베이스를 외부 DB가 아니라 내 로컬 환경에서 가볍게 쓸 수 있도록 SQLite와 Drizzle ORM으로 바꿔주고, 디자인은 Tailwind CSS를 써서 모던한 '다크 테마'로 싹 고쳐줘.
```

놀라운 점은 코딩 퍼포먼스였습니다. 기존 프로젝트 코드를 던져주자마자 Drizzle ORM이라는 새로운 기술 스택으로 데이터베이스 구조를 다시 짜고, 7만 건 이상의 대용량 CSV 파일을 통째로 올리면 서버가 터질 수 있다는 것을 고려해 **'청크 단위(Chunk) 쪼개기 업로드 기능'**과 다크 테마 UI를 매끄럽게 연결해 버렸습니다. 전체 숲(Context)을 보는 이해도가 엄청났습니다.

---

### 2단계: "오!" 소리가 절로 나온 스스로 디버깅

위풍당당하게 마이그레이션이 끝났다고 생각해 대용량 CSV 파일을 넣고 업로드 버튼을 눌렀습니다. 그런데 백엔드에서 멈추며 에러를 뱉어냈습니다. 바로 `NOT NULL constraint` 에러였습니다.

```text
SQLite로 변환 후 CSV 파일 업로드를 테스트해보니 데이터 적재가 실패해.
uploadedAt 필드에 NOT NULL 제약 조건 에러가 발생하는데, 원인을 파악하고 고쳐줘.
```

저는 오류 로그만 툭 던졌는데, Antigravity가 스스로 프런트엔드의 업로드 컴포넌트(`CsvUploader`)와 백엔드 API 라우터(`routers.ts`) 중간에서 데이터 매핑이 누락된 지점을 정확히 짚어냈습니다. 단순히 코드 수정 한 줄을 알려주는 게 아니라, **"프런트엔드에서 데이터가 넘어올 때 빈 값을 쏘고 있으니, 백엔드에서 그것을 방어하고 동적으로 서버時間を 할당하도록 보완했습니다"**라며 구조적인 치료까지 마쳤습니다. 에러 찾으려고 땀 빼던 시절을 생각하니 정말 감탄이 나왔습니다.

## ✅ 결과 (After)

이 과정을 통해 제 처음 가설은 **성공적으로 검증**되었습니다.

### Before vs After

| 핵심 비교 항목 | 기존 환경 (단일 AI 의존) | 개선 방식 (Manus + Antigravity 분산) |
|------|--------|-------|
| **토큰/비용 효율** | 유지보수 및 자잘한 수정에도 상당히 높은 토큰 소모 발생 | 로컬 기반의 Antigravity로 가볍게 디버깅하여 체감 소모량 대폭 절감 |
| **운영 환경** | 외부 플랫폼 접속 및 종속성 존재 (데이터 업로드 부담) | 로컬 SQLite 기반의 가볍고 쾌적한 나만의 시스템 완성 (데이터 유출 걱정 0) |
| **아키텍처 확장성** | 단순히 돌아가는 초기 수준의 구조 | Drizzle ORM + 다크 테마 기반의 모던 스택 장착 완료 |

이 마이그레이션으로 인해 앞으로 대용량 데이터 시트(7만 행 이상)를 분석하거나 다룰 때 외부 클라우드를 신경 쓰지 않고 제 컴퓨터에서 마음껏 굴려볼 수 있는 탄탄한 환경을 얻었습니다.

## 💬 이 과정에서 배운 핵심 AI 활용 팁

### 💡 "AI 도구는 목적에 맞게 쪼개어(분산해서) 써보세요!"

많은 분들이 "가장 완벽한 AI 하나"를 찾아 모든 것을 맡기려 합니다. 하지만 경험상 AI마다 특화된 장점이 다릅니다. 

**프롬프트 하나로 전체 틀이나 웹앱의 초기 뼈대를 와일드하게 잡는 데에는 Manus를 활용하고,** 
**그 이후 특정 기능을 내 로컬 환경에 최적화시키며 치밀하게 디버깅하고 깎아나가는 과정은 Antigravity로 진행하세요.**

이렇게 목적에 따라 AI를 분산해서 사용하면, **토큰(돈)은 아끼면서 결과물의 퀄리티와 개발 효율은 2배 이상 치솟는 경험**을 할 수 있습니다.

## 🚀 앞으로의 계획

종속성 없고 언제든 만질 수 있는 제 로컬 앱 환경을 확보했으니, 앞으로는 정기 리포트 자동 생성이나 회사 메일망과 연동된 '주간 광고 성과 자동 발송 파이프라인' 등 사내 환경과 찰떡같이 붙어 돌아가는 추가 자동화 아이템들을 계속해서 만들어 볼 계획입니다.

## 📋 재사용 가능한 프롬프트

### 프롬프트 1: 초기 개발된 코드를 로컬 스택으로 마이그레이션(최적화) 할 때
> 기존에 작업 완료된 [프로젝트/코드베이스]를 바탕으로, 내 로컬 환경에서 더 가볍게 돌아가도록 스택을 변경해 줘.
> 데이터베이스는 별도 접속이 필요 없는 [SQLite]로 고치고 연관된 ORM(예: Drizzle) 세팅을 업데이트해 줘. 
> 디자인 컨셉은 전체적으로 [다크 테마]로 통일감 있게 엮어줘.

### 프롬프트 2: 에러 로그 피드백 기반 자동 치유를 요구할 때
> [기능명, 예: CSV 파일 업로드]를 테스트해보니 [에러 메시지 혹은 터진 현상, 예: NOT NULL constraint] 에러가 발생하며 멈췄어. 
> 프런트엔드 [컴포넌트/파일]와 백엔드 [API 라우터/DB] 간 데이터가 정상적으로 오고 가는지 구조적인 원인을 찾아서 고쳐줘.
